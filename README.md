# Практическая работа 4.1 — вариант 19

---

## 1. Цель работы

Освоить связку **Apache Airflow + dbt (data build tool)** для оркестрации трансформаций данных: подготовить окружение, загрузить сырые данные в СУБД, оформить dbt-проект (staging и витрины), описать **DAG** с последовательностью **`dbt run` → `dbt test`**, запустить и отмониторить выполнение, зафиксировать результаты в отчёте.

---

## 2. Выбранный вариант выполнения

Реализован **вариант 2 — локальное решение (Docker)** из методички: Airflow, PostgreSQL и сопутствующие сервисы развёрнуты через **Docker Compose** на **Ubuntu**.

**Обоснование выбора:** не требуется облачный аккаунт и биллинг; полный контроль над окружением; быстрая отладка DAG и dbt; достаточный объём RAM на машине (рекомендуется от 4 GB для комфортной работы стека, как в методичке).
---

## 3. Краткое описание решения (архитектура ELT)

1. **Extract / Load:** исходные **CSV** (датасет по варианту) загружаются в **PostgreSQL**, схема **`raw_data`** (таблицы по файлам датасета).

2. **Transform:** в **dbt** описаны модели **`staging`** (представления поверх `source`) и **`marts`** (витрины, в т.ч. **`mart_department_budget`** и при необходимости дополнительные модели, например `budget_performance`). Конфигурация: `dbt_project.yml`, `models/staging/_sources.yml`, тесты для витрины.

3. **Orchestrate:** **Airflow DAG** (`dbt_company_budget_v19`) из двух задач на **`BashOperator`**: **`dbt_run`** затем **`dbt_test`** (зависимость **`dbt_run >> dbt_test`**), как требует методичка (шаг 4). Путь к проекту в контейнере: **`/opt/airflow/dbt/my_dbt_project`**.

Итог: повторяемый ELT-конвейер «сырые данные в PostgreSQL → трансформации dbt → проверка тестами под управлением Airflow».

## 6. Инструкция по запуску и проверке

Ниже — минимальный порядок действий; детали см. в блокноте с инструкцией (части **1–5**).

1. Установить **Docker** и **Docker Compose**, подготовить каталог проекта (`~/pr41_budget`).

2. Собрать образ и поднять стек:
   ```bash
   cd ~/pr41_budget
   docker compose build
   docker compose up -d
   docker compose ps   # все нужные сервисы healthy
   ```

3. Создать схему **`raw_data`** и загрузить CSV (часть **3**), например через `docker compose exec postgres psql ...` и `COPY`.

4. Проверить dbt с хоста (часть **4**): виртуальное окружение, `dbt debug`, `dbt run`, `dbt test` (профиль с **`localhost`** и внешним портом Postgres, если он не стандартный).

5. Убедиться, что в **`dags/`** размещён DAG с **`dbt_run >> dbt_test`** и путём **`DBT_PROJECT_DIR=/opt/airflow/dbt/my_dbt_project`**.

6. Открыть **веб-интерфейс Airflow** (для Docker по методичке: `http://localhost:8080` или `http://<IP_ВМ>:8080`), найти DAG **`dbt_company_budget_v19`**, **активировать** и **запустить вручную** (шаг 5 методички). Убедиться в успехе задач **`dbt_run`** и **`dbt_test`**.

7. В **PostgreSQL** проверить наличие витрин

## 8. Выводы

В ходе работы были использованы **Airflow** для планирования и мониторинга конвейера и **dbt** для декларативных SQL-трансформаций и тестов качества. Такой подход отделяет оркестрацию от логики трансформаций и упрощает повторяемость развёртывания.

**Сравнение с альтернативой (GCP):** локальный Docker не требует настройки биллинга и облачной инфраструктуры и удобен для учебной отладки; облачный вариант (Cloud Composer, BigQuery) ближе к промышленным сценариям масштабирования и управляемым сервисам, но сложнее по доступу и стоимости.
